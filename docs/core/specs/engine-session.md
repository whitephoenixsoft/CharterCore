# ENG-SESSION — Session Lifecycle, Governance Orchestration & Receipt Emission

Status: REFACTORED (v13 – Reference-Driven, Responsibility-Aligned)  
Applies to: Engine Core (V1/V2+)  

Authority: Subordinate to ENG-DOMAIN, ENG-DECISION, ENG-RECEIPT, ENG-INTEGRITY, ENG-SUPERSESSION, ENG-REVIEW-RETIRED, ENG-CANON, ENG-SPECVERIFY

---

# 1. Purpose

## ENG-SESSION-01 — Sessions Are the Sole Legitimacy Mechanism

Sessions are the only mechanism through which legitimacy may be created.

All:

- Resolution acceptance  
- Resolution supersession  
- Authority evolution  
- Scope evolution  

must occur through a valid session acceptance.

Receipts formalize terminal closure but do not create legitimacy.

Fail if:

- A Resolution is created outside a session  
- Legitimacy emerges without session acceptance  
- Supersession occurs without a session  

---

# 2. Session Classification & Bootstrap Constraints

## ENG-SESSION-02 — Session Type Is Explicit and Immutable

Each session is classified at creation as exactly one of:

- AUTHORITY  
- SCOPE  
- REGULAR  

Session type:

- Required at creation  
- Immutable for the lifetime of the session  
- Determines governance preconditions  

Fail if:

- Session type changes  
- Session type undefined  

---

## ENG-SESSION-03 — Creation Preconditions

Governance preconditions must be validated before session creation.

These rules are defined in:

- ENG-DECISION (governance evaluation)  
- ENG-SUPERSESSION (ACTIVE derivation)  
- ENG-INTEGRITY (slot validity)  

Summary:

### AUTHORITY
- Allowed if no ACTIVE Authority exists OR valid supersession applies  

### SCOPE
- Requires exactly one ACTIVE Authority  

### REGULAR
- Requires exactly one ACTIVE Authority and one ACTIVE Scope  

Fail if governance invariants are violated.

---

# 3. Identity & Participation Epochs

## ENG-SESSION-04 — Session Identity

- session_id must be UUIDv7 (ENG-DOMAIN)  
- Generated by the Engine  
- Immutable  

---

## ENG-SESSION-04A — Participant Epochs

Participant identity represents a participation epoch.

Rules:

- UUIDv7, session-scoped  
- Never reused  
- Bound to exactly one round  

Epoch semantics and validation defined in:

- ENG-DOMAIN  
- ENG-INTEGRITY  

---

## ENG-SESSION-04B — Candidate Epochs

Candidate identity represents a proposal epoch.

Rules:

- UUIDv7, round-scoped  
- Unique within round  
- Never reused across rounds  

Canonical structure defined in ENG-DOMAIN.

---

# 4. Session State & Phase Model

## ENG-SESSION-04C — State & Phase Consistency

Session state and phase must remain consistent.

States:

- ACTIVE  
- PAUSED  
- BLOCK_TEMPORARY  
- BLOCK_PERMANENT  
- ACCEPTED  
- CLOSED  

Phases:

- PRE_STANCE  
- VOTING  
- TERMINAL  

Consistency rules:

- TERMINAL ↔ {ACCEPTED, CLOSED}  
- VOTING requires vote presence  
- PRE_STANCE requires no votes  

Validation enforced via:

- ENG-ERROR (EvaluationReport)  
- ENG-INTEGRITY (structural consistency)  

---

# 5. Round Model

## ENG-SESSION-05 — Deterministic Round Segmentation

- Sessions begin at round_index = 1  
- Rounds are contiguous and monotonic  
- New rounds created only via RESUME  

On RESUME:

- All round-scoped collections cleared  
- round_index increments  
- phase resets to PRE_STANCE  

Round snapshot representation is defined in:

- ENG-RECEIPT  

---

# 6. Governance Context Snapshot

## ENG-SESSION-06 — Immutable Governance Snapshot

At session creation:

- Authority Resolution snapshot captured  
- Scope Resolution snapshot captured  

Snapshots are immutable.

ENG-SESSION records governance snapshots and provides them for evaluation.

Usability and validity of referenced artifacts are determined externally by:

- ENG-REVIEW-RETIRED  
- ENG-SUPERSESSION  
- ENG-DECISION  

---

# 7. Freeze Boundary

## ENG-SESSION-07 — PRE_STANCE Mutability Boundary

PRE_STANCE is the only mutable phase.

Mutable:

- Participants  
- Candidates  
- Constraints  

Transition to VOTING freezes all sets.

Triggered by:

- First vote  
- Solo mode implicit vote  

Mutation boundaries are structurally defined in ENG-SESSION.

Violations are evaluated and enforced during decision evaluation (ENG-DECISION) and reported via ENG-ERROR.

---

# 8. Vote Rules

## ENG-SESSION-08 — Mechanical Voting

Rules:

- One vote per participant per candidate per round  
- Votes are immutable once recorded  
- Votes are bound to a single round  

Vote structure defined in ENG-DOMAIN.  
Evaluation semantics defined in ENG-DECISION.

---

## ENG-SESSION-08A — Solo Mode

If exactly one participant exists:

- An implicit ACCEPT vote is applied during evaluation  

Behavior is defined in ENG-DECISION.  
ENG-SESSION does not create vote artifacts implicitly.

---

# 9. Resume & Epoch Reset

## ENG-SESSION-09 — Deterministic Reset

On RESUME:

- All participant epochs terminate  
- All round-scoped data is cleared  
- A new round is created  
- round_index increments  
- phase resets to PRE_STANCE  

Epoch integrity is enforced by:

- ENG-INTEGRITY  
- ENG-DOMAIN  

---

# 10. Acceptance Invocation

## ENG-SESSION-10 — Acceptance Is Invoked, Not Defined, by Session

The session layer is responsible for invoking acceptance.

It does not define acceptance eligibility or validation rules.

Acceptance eligibility is determined exclusively by:

- ENG-DECISION → eligibility determination  
- ENG-INTEGRITY → structural validity  
- ENG-SUPERSESSION → graph validity and conflict detection  
- ENG-REVIEW-RETIRED → usability constraints  

ENG-SESSION must:

- pass the current session state to the decision layer  
- remain non-mutating if eligibility fails  
- invoke atomic commit if eligibility succeeds  

---

## ENG-SESSION-10A — Acceptance Preconditions Are External

The session layer must not independently determine:

- governance validity  
- authority rule satisfaction  
- constraint satisfaction  
- supersession validity  

These are defined and evaluated by authoritative specifications and consumed by the session layer.

---

## ENG-SESSION-10B — Acceptance Commit Delegation

If acceptance eligibility succeeds:

- Session transitions to ACCEPTED  
- Atomic commit is executed as defined in ENG-PERSISTENCE  
- Resolution is created  
- Supersession updates are applied  
- Receipt is emitted  

ENG-SESSION must not:

- perform partial mutation  
- define commit ordering  
- redefine atomic guarantees  

---

# 11. Receipt Emission

## ENG-SESSION-11 — Terminal Receipt Requirement

Each session must emit exactly one receipt.

Types:

- ACCEPTED → LEGITIMACY  
- CLOSED → EXPLORATION  

Receipt structure and canonical serialization defined in:

- ENG-RECEIPT  
- ENG-CANON  

Rule identity binding defined in:

- ENG-SPECVERIFY  

Receipt emission occurs within the atomic commit boundary defined in ENG-PERSISTENCE.

---

# 12. Area Blocking Awareness

## ENG-SESSION-12 — Acceptance Must Respect Blocking Conditions

Session acceptance must respect Area-level and session-level blocking conditions.

Blocking determination is defined in:

- ENG-DECISION  
- ENG-REVIEW-RETIRED  
- ENG-INTEGRITY  

ENG-SESSION must not allow acceptance if blocking conditions apply.

---

# 13. Immutability

## ENG-SESSION-13 — Terminal Immutability

If session is terminal:

- No further mutations are permitted  

Enforced via:

- ENG-INTEGRITY  
- ENG-ERROR  

---

# 14. Audit

## ENG-SESSION-14 — Audit Emission

Lifecycle events must emit audit records.

Audit rules defined in:

- ENG-AUD  

Audit is observational only and must not affect legitimacy or evaluation.

---

# 15. Failure Semantics

## ENG-SESSION-15 — Deterministic Failure

All failures:

- Must be explicit  
- Must not mutate state  
- Must produce an EvaluationReport  

Defined in:

- ENG-ERROR  

---

# 16. Summary Guarantees

- Sessions are the sole mechanism for legitimacy creation  
- Session type is explicit and immutable  
- Governance snapshots are immutable  
- PRE_STANCE is the only mutation phase  
- Rounds are deterministic and isolated  
- Acceptance eligibility is externally defined and consumed  
- Atomic commit is enforced via ENG-PERSISTENCE  
- Receipts are canonical via ENG-RECEIPT and ENG-CANON  
- Supersession is enforced via ENG-SUPERSESSION  
- Usability is enforced via ENG-REVIEW-RETIRED  
- Structural integrity is enforced via ENG-INTEGRITY  

---

# 17. Mental Model

Sessions orchestrate legitimacy.

They define:

- lifecycle  
- participation structure  
- round structure  
- governance snapshot capture  

They do not define:

- structural integrity  
- supersession logic  
- usability semantics  
- acceptance eligibility  
- canonical serialization  

They invoke these through authoritative specifications.

Acceptance is the coordinated outcome of:

- structural validation  
- governance validation  
- authority rule evaluation  
- constraint validation  
- supersession validation  
- atomic persistence  
- receipt emission  

Nothing is inferred.  
Everything is delegated.  
Legitimacy is created only at atomic acceptance.
