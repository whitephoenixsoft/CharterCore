# ENG-SESSION — Session Lifecycle, Governance Orchestration & Receipt Emission

Status: REFACTORED (v12 – Reference-Driven Model)  
Applies to: Engine Core (V1/V2+)  

Authority: Subordinate to ENG-DOMAIN, ENG-DECISION, ENG-RECEIPT, ENG-INTEGRITY, ENG-SUPERSESSION, ENG-REVIEW-RETIRED, ENG-CANON, ENG-SPECVERIFY

---

# 1. Purpose

## ENG-SESSION-01 — Sessions Are the Sole Legitimacy Mechanism

Sessions are the only mechanism through which legitimacy may be created.

All:

- Resolution acceptance
- Resolution supersession
- Authority evolution
- Scope evolution

must occur through a valid session acceptance.

Receipts formalize terminal closure but do not create legitimacy.

Fail if:

- A Resolution is created outside a session
- Legitimacy emerges without session acceptance
- Supersession occurs without a session

---

# 2. Session Classification & Bootstrap Constraints

## ENG-SESSION-02 — Session Type Is Explicit and Immutable

Each session is classified at creation as exactly one of:

- AUTHORITY
- SCOPE
- REGULAR

Session type:

- Required at creation
- Immutable for the lifetime of the session
- Determines governance preconditions

Fail if:

- Session type changes
- Session type undefined

---

## ENG-SESSION-03 — Creation Preconditions

Governance preconditions must be validated before session creation.

These rules are defined in:

- ENG-DECISION (governance evaluation)
- ENG-SUPERSESSION (ACTIVE derivation)
- ENG-INTEGRITY (slot validity)

Summary:

### AUTHORITY
- Allowed if no ACTIVE Authority exists OR valid supersession applies

### SCOPE
- Requires exactly one ACTIVE Authority

### REGULAR
- Requires exactly one ACTIVE Authority and one ACTIVE Scope

Fail if governance invariants are violated.

# 3. Identity & Participation Epochs

## ENG-SESSION-04 — Session Identity

- session_id must be UUIDv7 (ENG-DOMAIN)
- Generated by the Engine
- Immutable

---

## ENG-SESSION-04A — Participant Epochs

Participant identity represents a participation epoch.

Rules:

- UUIDv7, session-scoped
- Never reused
- Bound to exactly one round

Epoch semantics and validation defined in:

- ENG-DOMAIN
- ENG-INTEGRITY

---

## ENG-SESSION-04B — Candidate Epochs

Candidate identity represents a proposal epoch.

Rules:

- UUIDv7, round-scoped
- Unique within round
- Never reused across rounds

Canonical structure defined in ENG-DOMAIN.

---

# 4. Session State & Phase Model

## ENG-SESSION-04C — State & Phase Consistency

Session state and phase must remain consistent.

States:

- ACTIVE
- PAUSED
- BLOCK_TEMPORARY
- BLOCK_PERMANENT
- ACCEPTED
- CLOSED

Phases:

- PRE_STANCE
- VOTING
- TERMINAL

Consistency rules:

- TERMINAL ↔ {ACCEPTED, CLOSED}
- VOTING requires vote presence
- PRE_STANCE requires no votes

Validation enforced via:

- ENG-ERROR (EvaluationReport)
- ENG-INTEGRITY (structural consistency)

---

# 5. Round Model

## ENG-SESSION-05 — Deterministic Round Segmentation

- Sessions begin at round_index = 1
- Rounds are contiguous and monotonic
- New rounds created only via RESUME

On RESUME:

- All round-scoped collections cleared
- round_index increments
- phase resets to PRE_STANCE

Round snapshot representation is defined in:

- ENG-RECEIPT (canonical structure)

---

# 6. Governance Context Freeze

## ENG-SESSION-06 — Immutable Governance Snapshot

At session creation:

- Authority Resolution snapshot captured
- Scope Resolution snapshot captured

Snapshots are immutable.

Acceptance must validate that referenced governance artifacts remain usable.

Usability rules are defined in:

- ENG-REVIEW-RETIRED (UNDER_REVIEW / RETIRED semantics)
- ENG-SUPERSESSION (SUPERSEDED semantics)

Fail if referenced artifacts are not usable at evaluation time.

---

# 7. Freeze Boundary

## ENG-SESSION-07 — PRE_STANCE Mutability Boundary

PRE_STANCE is the only mutable phase.

Mutable:

- Participants
- Candidates
- Constraints

Transition to VOTING freezes all sets.

Triggered by:

- First vote
- Solo mode implicit vote

Mutation rules defined in:

- ENG-DECISION (governance mutability)
- ENG-ERROR (freeze violations)

---

# 8. Vote Rules

## ENG-SESSION-08 — Mechanical Voting

Rules:

- One vote per participant per candidate per round
- Votes are immutable
- Votes bound to round

Vote structure defined in ENG-DOMAIN.  
Evaluation semantics defined in ENG-DECISION.

---

## ENG-SESSION-08A — Solo Mode

If one participant exists:

- Implicit ACCEPT vote inserted during evaluation

Behavior defined in ENG-DECISION.

---

# 9. Resume & Epoch Reset

## ENG-SESSION-09 — Deterministic Reset

On RESUME:

- Participant epochs terminate
- All round data cleared
- New round created

Epoch integrity enforced by:

- ENG-INTEGRITY
- ENG-DOMAIN

---

# 10. Deterministic Acceptance

## ENG-SESSION-10 — Acceptance Orchestration

Acceptance is a deterministic orchestration process.

The session layer does not define validation rules; it invokes:

- ENG-INTEGRITY → structural validation & halt conditions  
- ENG-SUPERSESSION → ACTIVE derivation & conflict detection  
- ENG-REVIEW-RETIRED → usability validation  
- ENG-DECISION → authority rule & constraint evaluation  

Requirements:

- Full validation pass
- No short-circuiting
- Deterministic output

Fail if any referenced object is not usable or constraints fail.

---

## ENG-SESSION-10A — Constraint Enforcement

Constraint semantics defined in ENG-DECISION.

Session must:

- Evaluate constraints deterministically
- Fail acceptance if constraints unmet

---

## ENG-SESSION-10B — Acceptance Transaction

Acceptance triggers atomic commit defined in:

- ENG-PERSISTENCE

Atomic boundary includes:

- Session → ACCEPTED
- Resolution creation
- Supersession updates
- Receipt emission

Resolution structure defined in ENG-DOMAIN.  
Supersession rules defined in ENG-SUPERSESSION.

---

# 11. Receipt Emission

## ENG-SESSION-11 — Terminal Receipt Requirement

Each session must emit exactly one receipt.

Types:

- ACCEPTED → LEGITIMACY
- CLOSED → EXPLORATION

Receipt structure and canonical serialization defined in:

- ENG-RECEIPT
- ENG-CANON

Rule identity binding defined in:

- ENG-SPECVERIFY

Receipt emission occurs within the atomic commit boundary (ENG-PERSISTENCE).

- No session may ACCEPT

Blocking rules defined in:

- ENG-DECISION
- ENG-REVIEW-RETIRED

# 16. Summary Guarantees

- Sessions are sole legitimacy mechanism
- Governance snapshot immutable
- PRE_STANCE is sole mutation phase
- Rounds deterministic and isolated
- Acceptance delegates to authoritative specs
- Atomic commit enforced via ENG-PERSISTENCE
- Receipts canonical via ENG-RECEIPT + ENG-CANON
- Supersession enforced via ENG-SUPERSESSION
- Usability enforced via ENG-REVIEW-RETIRED
- Structural integrity enforced via ENG-INTEGRITY

---

# 17. Mental Model

Sessions orchestrate legitimacy.

They do not define:

- Structural integrity
- Supersession logic
- Usability semantics
- Canonical serialization

They invoke them.

Acceptance is a coordination of:

- Integrity validation
- Supersession validation
- Governance usability
- Constraint evaluation
- Atomic persistence
- Receipt emission

Nothing is inferred.  
Everything is delegated.  
Legitimacy is created only at atomic acceptance.
